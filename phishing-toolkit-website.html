// process

Three checks, one verdict

The console doesn't guess — it runs the same input through a fixed set of pattern checks used in real phishing triage, then totals a risk score.

01

Parse

The URL or email is broken into parts — domain, subdomains, sender address, reply-to, embedded links, body text.

02

Match

Each part is checked against known phishing patterns: typosquatted domains, mismatched reply addresses, urgency language, raw IP hosts.

03

Score & explain

Every match adds to a risk score. Nothing is hidden — the log shows exactly which signals fired and why they matter.

What it actually looks for

Domain impersonation & typosquatting

Compares domains against known brand names using edit-distance matching — catches things like "paypa1.com" or "secure-paypal-verify.ru".

Sender / Reply-To mismatch

Legitimate companies rarely route replies to a different domain than they send from. A mismatch is a strong signal of redirection.

Urgency & pressure language

Phrases like "verify now," "account suspended," or "24 hours" are designed to short-circuit careful reading.

Structural red flags

Raw IP addresses instead of domains, excessive subdomains, unencrypted HTTP, and disposable TLDs (.tk, .ru, .xyz) used in throwaway phishing infrastructure.

Direct credential requests

Legitimate services almost never ask you to type a password or card number directly into an email-linked page.

Limitations & honest tradeoffs

  • Rule-based, not ML-based. It catches known patterns reliably but will miss novel phishing techniques that don't match any rule — a trained classifier on a labeled dataset would generalize better.
  • No live infrastructure checks. It doesn't query WHOIS, domain age, SSL certificate issuers, or SPF/DKIM/DMARC records — all of which a production tool would check.
  • Keyword lists are a starting point. Urgency-language and brand-keyword lists are small and English-only; attackers iterate constantly.
  • False positives are possible. Legitimate marketing emails sometimes use urgency language too ("sale ends tonight") — the score is a triage aid, not a final verdict.

What I'd add with more time

  • SPF/DKIM/DMARC header validation for real authentication checks
  • Domain age + WHOIS lookup (newly registered domains are a strong signal)
  • A small trained classifier on a labeled phishing/legitimate email dataset, compared against this rule-based baseline
  • Browser extension version for real-time inbox scanning
